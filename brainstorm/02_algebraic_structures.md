# 02 — Algebraic Structures Beyond SO(n) / Heisenberg

Brainstorm angle: new associative structures for parallel-scan RNN cells, ranked by code-LLM relevance after 2024–2026 lit search. Calibration against what we already shipped (`SO(n)`, `H_3`, `H_d`, `U_4`, `Magnus K=2`, `Dyck`, `GF(2)-Heisenberg`, `Sig K=3`) and against the published competition (DeltaProduct, PD-SSM, PaTH, AUSSM, KLA, Kimi Linear, SLiCEs, Tropical Attention, AdS-GNN).

The big constraint our killgate already proved: a single cell can hit a parity wall, but if it doesn't survive on real text the win is wasted. So the candidates below are filtered for *plausibly useful on code, not just on synthetic group word problems* — bracket nesting, scope/type tracking, cross-reference, mod-p control flow.

## Top 5 concrete ideas

### Idea 1: SL(2, ℝ)-cell ("multi-d Möbius / hyperbolic Iwasawa")

- **What it is:** State per head is a 2×2 real matrix in `SL(2, ℝ)`; combine is matrix multiplication. Acts on an `H²` half-plane via `z ↦ (az+b)/(cz+d)`. Iwasawa decomposition `KAN` separates rotation (compact `K = SO(2)`), dilation (`A = diag(λ, 1/λ)`), and translation (`N`) — three input-dependent generators per token, parameter-efficient. Multi-head version stacks several `SL(2)` chains, mixed via channel projection.
- **Parallel-scan story:** 2×2 matrix product is associative, 8-flop combine, vectorises well in Triton (`m32 × m32` warpgroup MMA on Blackwell). Per-head state cost = 4 scalars vs 2 for diagonal SSM, 16 for SO(4), `d²` for Heisenberg.
- **Expressivity:** SL(2, ℝ) eigenvalues live on `ℝ \ {0}` not `[0,1]`, so it dodges Grazzi's TC⁰ wall *without* needing complex numbers. Hyperbolic dynamics give built-in O(1) state long-range attractor that Mamba's `(0,1)^d` decay cannot. The compact subgroup `K` does parity. The Borel subgroup `B = AN` does scale/translation — exactly the affine state Kalman Linear Attention exploits, with KLA being the special diagonal case.
- **Key papers:** [KLA (Shukla 2026, arXiv:2602.10743)](https://arxiv.org/abs/2602.10743) (1-D Möbius, abelian); [MöbiusAttention (Deng 2024, arXiv:2409.12175)](https://arxiv.org/abs/2409.12175) (Möbius as static attention, *not as scan*); [Lorentzian ResNet (Mishne 2024, arXiv:2412.14695)](https://arxiv.org/abs/2412.14695) (hyperbolic, not as scan).
- **Implementation cost:** S. Triton kernel is a tiny modification of `kernels/ortho_son/` — same matmul-scan structure but on 2×2 (per head) instead of n×n.
- **Code-LLM relevance:** medium. Hyperbolic dilation matches "scope opens / scope closes" exponential nesting growth; the boost generator naturally encodes "indentation level" without integer counters. The `K` factor handles parity-of-brackets. Rich enough to learn variable-renaming without recall.
- **Risk:** SL(2, ℝ) is solvable, so by Krohn-Rhodes it gives only NC¹-with-counters expressivity, not the non-solvable hardness of S₅. If the bottleneck on code turns out to need non-solvable symmetry composition (it probably doesn't), this is too weak. Numerical drift on long T (det → 0 instabilities) — mitigate by parameterising via Iwasawa generators rather than free 2×2 entries.

### Idea 2: GL(2, GF(p)) discrete cell ("modular projective")

- **What it is:** State per head is a 2×2 matrix over `GF(p)` for small prime `p ∈ {3, 5, 7, 11}`; combine is matrix multiplication mod `p`; differentiable surrogate via straight-through softmax over the `(p-1) p (p²-1)` group elements, or a learned ℝ-valued approximation discretised at readout. `GL_2(GF(p))` is *non-solvable* for `p ≥ 5` (it contains `PSL_2(GF(p)) ≅ A_5` for `p = 5`).
- **Parallel-scan story:** Group-multiplication, same combine pattern as PD-SSM but in a richer non-abelian group. With `p=5`, state size ≈ `log₂(480) ≈ 9` bits per head — almost free.
- **Expressivity:** `PSL_2(GF(p))` is the smallest interesting non-solvable simple group family. By Krohn-Rhodes, having one of these as a primitive lifts NC¹-without-it to NC¹-with-it, which is the boundary that DeltaProduct and PaTH push against using Householder/SL(d) products. The win vs DeltaProduct: DeltaProduct uses `O(d)` Householders to *approximate* arbitrary group composition; we use the actual finite simple group natively, with much smaller state.
- **Key papers:** [DeltaProduct (Siems 2025, arXiv:2502.10297)](https://arxiv.org/abs/2502.10297) (Householder products, hits S₅ via `K=4` layers); [PD-SSM (IBM, NeurIPS '25 spotlight, arXiv:2509.22284)](https://arxiv.org/abs/2509.22284) (permutation × diagonal, gets *all* solvable FSAs but not non-solvable); [Grazzi et al. (ICLR '25, arXiv:2411.12537)](https://arxiv.org/abs/2411.12537) (negative eigenvalues).
- **Implementation cost:** M. The discrete forward is trivial (table lookup). Differentiability is the hard part — pick *one* of: (a) learned ℝ-valued surrogate that lives in `M_2(ℝ)` and projects to nearest `GL_2(GF(p))` at inference; (b) Gumbel-softmax over the finite group; (c) embed `GL_2(GF(p))` into `GL_2(ℝ)` via the integer-matrix lift, train with continuous combine, snap at eval.
- **Code-LLM relevance:** medium-high. Mod-p control-flow patterns (loop counters, hash-table indexing, parity-of-brackets) sit naturally in `GF(p)`. More importantly: proves we can do non-solvable state-tracking with *constant* (≈ 9-bit) state per head, vs DeltaProduct's `O(d)`.
- **Risk:** Discrete-output gradient is the standard bugbear. The straight-through estimator for finite-group composition is likely high-variance. If the surrogate is `M_2(ℝ)` we collapse to "just another matrix-state cell" with no advantage. Also, the user wants *coding LLM wins*, and there's no published evidence that real Python-LM perplexity is bounded by an `S_5`-shaped task; the killgate may not fire.

### Idea 3: Clifford / multivector geometric algebra cell

- **What it is:** State is a multivector in `Cl(p, q, 0)` for small `(p,q)` — e.g. `Cl(3,0)` (8-dim, projective 3D rotor), `Cl(3,1)` (16-dim spacetime / GATr), or `Cl(2,0,1)` (PGA, 8-dim conformal-like). Combine is the geometric product (associative, non-commutative). Each token emits a per-head multivector; scan multiplies them.
- **Parallel-scan story:** Geometric product is matrix-multiplication-shaped on the multivector basis (16×16 for `Cl(3,1)` is a single `mma m16n16k16`). Sparse-multivector tricks reduce flops by 4-8× (most tokens emit grade-1 or grade-2 only).
- **Expressivity:** Multivector product *contains* quaternion product, complex product, Pauli matrices, dual numbers, and the spinor representation of Lorentz `SO(3,1)`. Specifically: rotors live in the even subalgebra and form `Spin(p,q)`, which is a *double cover* of the rotation group — eigenvalues on the unit circle (Grazzi-clean) plus reflections (parity-clean) in one cell. The `Cl(3,1)` even subalgebra contains both `SU(2)` (compact rotations) and `SL(2, ℂ)` (boosts) as subgroups, so it superset both Idea 1's hyperbolic bit and a non-abelian rotation-with-flip.
- **Key papers:** [GATr (Brehmer et al., NeurIPS '23)](https://arxiv.org/abs/2305.18415); [CliffordNet (2026, arXiv:2601.06793)](https://arxiv.org/abs/2601.06793) (vision backbone *with* O(N) sparse rolling — a direct precedent for "scan via geometric product"); [Versor sequence model (2026, arXiv:2602.10195)](https://arxiv.org/abs/2602.10195) (multi-channel Clifford for sequence). None of these uses the geometric product as the *associative scan combine* of a causal LM cell — it's always either a token mixer or an equivariant linear layer.
- **Implementation cost:** M. We have skeleton matmul scans; the multivector basis tables are fixed and small. Most engineering effort is choosing `(p,q)` and writing the readout (multivector → ℝ^d for the residual).
- **Code-LLM relevance:** medium. The "rotor scan" angle gives us the SO(n) wins for free. Ungraded multivector lets the model learn its own algebra basis, including reflection (`GF(2)` parity) and dual quantities (Jet/dual-number-style first-derivatives — useful for tracking variable mutation across edits). Real reason for medium not high: code is structural and discrete; geometric algebra's home turf is continuous physics.
- **Risk:** Hot research area, lots of GATr/CliffordNet variants in the air. Differentiation from CliffordNet (which already does sparse rolling) needs to be sharp — our angle is "geometric product as the *causal scan combine* in an LM, replacing DeltaNet's outer product".

### Idea 4: Stack / inverse-monoid scan with pop tokens (context-free reach)

- **What it is:** State is the symbolic representation of a stack height + top-of-stack content, but with an *associative* combine. Trick: encode state as a finitely-supported element of the *bicyclic monoid* `B = ⟨a, b | ab = 1⟩` (where `a = push`, `b = pop`, `ab = id`, `ba ≠ id`). Every element has a unique normal form `b^m a^n` (`m` left-pops, `n` un-popped pushes). Combine: `(b^{m₁} a^{n₁}) · (b^{m₂} a^{n₂}) = b^{m₁ + max(m₂ - n₁, 0)} a^{n₂ + max(n₁ - m₂, 0)}` — exactly our `Dyck.lean` monoid `(l, r)`, but extended with content carried on each push (the top-of-stack symbol).
- **Parallel-scan story:** Already proven associative in `StateDep/Dyck.lean`. Adding "content per push" turns it into an inverse semigroup of partial bijections; combine is O(state-size), but the state can be unbounded. Bounded approximation: keep only the top-K stack entries (FIFO drop on overflow). With K=8 and 32-dim symbol per push, state is 256 floats.
- **Expressivity:** This is the bracket-balance *plus* matched-content-recall primitive, which neither DeltaNet (single-state) nor PD-SSM (finite state) provides. Specifically, recognises Dyck-`k` (matched brackets, `k` types) for `k ≤ K` — context-free, beyond the regular ceiling of all current parallel-scan SSMs. [Hewitt et al. EMNLP '20](https://arxiv.org/abs/2010.07515) gives the `O(m log k)` memory lower bound; our monoid lives in this minimum.
- **Key papers:** [Hewitt et al. (2020)](https://arxiv.org/abs/2010.07515) (Dyck memory bound); [Stack Attention (DuSell ICLR '24)](https://openreview.net/pdf?id=XVhm3X8Fum) (stack-augmented transformer, *not associative*); [Merrill & Sabharwal "Illusion of State" (ICML '24, arXiv:2404.08819)](https://arxiv.org/abs/2404.08819) (SSMs are TC⁰, can't do Dyck-2). Inverse semigroups in pushdown-monoid form: [Step et al. SuperdeterministicPDA monoid construction (2016)](https://link.springer.com/chapter/10.1007/978-3-662-53132-7_32). No NN paper combines an *associative* stack monoid with content-recall.
- **Implementation cost:** M. We already have `Dyck.lean`. Extending to per-position content (a small ring buffer of top-K stack entries) is ~50 lines of PyTorch + a custom Triton scan kernel that cancels/queues entries.
- **Code-LLM relevance:** **high**. This is the structural primitive for matching parens, brackets, braces, BEGIN/END, indentation in/dedent — every code formatter is built around exactly this monoid. Strong candidate for the killgate proving "synthetic group win → real coding win".
- **Risk:** The bicyclic monoid is *not* a group (has zero divisors / "drops" of stack content). Differentiability of the truncation is what kills naive implementations — needs a soft-pop mechanism that doesn't quietly leak an exponentially-large gradient. Also: Transformers already do bracket matching reasonably well via copy heads; the bar is whether we beat them on *deep* nesting where their depth-`O(log n)` constraint bites.

### Idea 5: Truncated polynomial composition `(ℝ[[x]]/x^K, ∘)`

- **What it is:** State is a formal power series `f(x) = a₀ + a₁x + ... + a_{K-1} x^{K-1}` (think K=4 or 8). Combine is *composition* `f ∘ g` truncated mod `x^K`. Identity is `x`. Composition is associative but non-commutative, and is the natural infinite-dimensional cousin of upper-triangular `U_K` matrices (Idea 5 ≅ "infinite Faà di Bruno coefficients").
- **Parallel-scan story:** Truncated composition is a polynomial in K coefficients of cost `O(K²)` or `O(K log K)`. For K=8 this is 64 multiplies — cheaper than a 16×16 matmul. The Faà di Bruno formula gives an explicit Lie group structure on `{f : f(0) = 0, f'(0) ≠ 0}` (the *Birkhoff group* of formal diffeomorphisms).
- **Expressivity:** Composition group is non-solvable, infinite-dimensional pro-p in disguise. The K-truncated quotient is a unipotent algebraic group — it strictly contains our `U_4` cell (which is Idea 5 at K=4 truncated to Heisenberg-like grade-2). Coefficient `a_k` of `f ∘ g ∘ ...` in the long composition is the `k`-th iterated Lie bracket / nested derivative, which is *exactly the order-k path-signature statistic* seen in our `Sig K=3` cell — but with a different (compositional, not multiplicative) algebra.
- **Key papers:** [Continuity of Formal Power Series Products in Nonlinear Control (Gray–Duffaut Espinosa, FoCM '22)](https://link.springer.com/article/10.1007/s10208-022-09560-0) (Chen-Fliess composition group as Lie group — *exactly the structure we need*, used in nonlinear control, never deployed in NN scan). The closest NN precedent we found was [Hypercomplex NN review (2025)](https://www.sciencedirect.com/science/article/pii/S2215016125004881) which mentions truncated formal-series algebras only obliquely.
- **Implementation cost:** M. Triton kernel is K×K small-matrix-mul-shaped (Faà di Bruno coefficients are Bell polynomials, cached at compile time).
- **Code-LLM relevance:** medium. Composition is exactly "function call / function definition with renaming" — the natural algebra for `f(g(h(x)))` chains in code. Scope chains, lexical environments, dataflow through pure function pipelines all have a composition-group flavour.
- **Risk:** This is the most speculative pick — no existing NN scan uses formal-series composition. Implementation will reveal whether the gradient through Faà di Bruno is well-behaved (likely yes, since the Lie algebra is just polynomial vector fields, but unproven). The risk vs payoff is asymmetric: low cost to try at K=4, large theoretical upside if the composition structure aligns with code's call-chain.

## Recommendation

**Pick 1 (highest priority): Idea 4 — content-bearing stack monoid.** It's the only candidate on this list with a *direct* match to a known code task (bracket / scope tracking) where current parallel-scan SSMs provably fail (Merrill–Sabharwal). We already have the abstract monoid in Lean (`Dyck.lean`); extending to content-recall and writing a Triton kernel is one weekend. The killgate is clean: train two 135M models, evaluate on `Dyck-3-with-naming` (existing synthetic) and Python LM PPL stratified by indentation depth. If hybrid stack-monoid + DeltaNet beats pure DeltaNet at depth ≥ 6, ship it.

**Pick 2 (companion): Idea 1 — SL(2, ℝ) with Iwasawa parameterisation.** Cheapest of the new structures (one 2×2 matmul per head), strict generalisation of the published KLA cell, and complementary to Pick 1: stack monoid handles discrete bracket matching, SL(2) handles continuous "depth as scalar" + parity. Combined `[stack, SL2, deltanet]` triple-stack mirrors our existing `[ortho, deltanet]` pattern but with two non-overlapping algebraic specialists. Drop-in for the existing kernels' scan loop.

Skip Idea 2 (`GL_2(GF(p))`) until/unless we have evidence that real-text PPL is bottlenecked by non-solvable state-tracking — the synthetic group-theoretic story alone is not enough (we already learned that). Skip Idea 3 (Clifford) for now: GATr/CliffordNet/Versor are crowding the tropical/geometric niche, and our angle would have to be the Clifford-product-as-scan-combine specifically, which is a small delta. Idea 5 is a bench-warmer — try if Picks 1 and 2 hit a wall.

## Open questions

1. **Stack-monoid + DeltaNet recall interference.** The stack monoid is a *partial* function (push/pop with cancellation) — does it interfere with DeltaNet's fixed-frame recall the way our SO(n) conjugation did? Our incompatibility theorem in `RESULTS.md` is about two-sided rotation; partial-function composition is one-sided, so it *should* compose better, but we haven't proved it.
2. **Differentiable surrogate for finite group products.** Idea 2's straight-through estimator is the dominant published approach (Gumbel-softmax, REINFORCE). Is there a smoother alternative that uses the algebraic structure of `GL_2(GF(p))` rather than treating it as an opaque finite set?
3. **Code-task killgate parity.** The 2026-04-25 Heisenberg killgate was on synthetic parity. We need a code-specific synthetic that actually moves the needle on real Python PPL. Candidates: matched-bracket-recall-with-naming (stack-monoid territory), tail-call dataflow (composition-group territory), variable-aliasing (inverse-semigroup territory).
4. **Where does real Python PPL break?** Without a per-token-position breakdown of pure DeltaNet's loss on Python, we're guessing which structural primitive is missing. A diagnostic study (loss vs depth, vs scope size, vs control-flow nesting depth) would tell us whether to spend cycles on Idea 4 or Idea 5.
5. **Multi-head SL(2, ℝ) coupling.** Per-head `SL_2` is just `KLA` per head. The interesting structure is *across* heads: do we need a `SO(n)`-style mixer to let different heads coordinate, or does pre/post linear projection suffice (as in DeltaNet)?
