# Verdict: the Working Memory recall saga

## Summary
The longest investigation in the project. WM has **one clean win** (saturated MQAR, +11.1 pp) and a long history of being **inert on real recall**. The full causal chain was traced: (1) gradient-disconnect — WM never got a recall gradient; (2) key-separability — cosine addressing can't separate near-identical keys; (3) the mechanism *was* fixed and validated to **100 % synthetic recall** with name-key addressing + copy readout; but (4) on **real code** the recurrence already solves value-recall (~89 %); and (5, 2026-06-15) v14 showed WM trains itself off (`copy_g → 0`, code-recall off=1.000 *measured*) — BUT the honest reason is our realistic recall data only reaches **N≤24**, below WM's validated **N≥48** niche, so off=1.0 is expected (the agentic "42 %→100 %" was partly inferred, not run). and (6, **2026-06-16 — OVERTURNS "drop WM"**) the off=1.000 was a **LEAK** (the eval scored the *restated* answer); leak-free, the recurrence scores **0%** on real `NAME=value` recall, and a **discrete-key** WM (hash the identifier token-span → exact match → copy readout) gives **100% recall, integrated end-to-end, Pareto-safe** (learned gate selective 0.9999/0.0011, base CE +0.0009). **UPDATED VERDICT: WM WORKS — keep it.** See the UPDATED VERDICT section below. Source: `project_wm_recall_probe_broken_and_routed_around.md`, `project_wm_addressing_root_cause.md`, `project_recall_discrete_key_direction.md`, `project_recall_*`.

## The arc (in order)
1. **MQAR win** (the existence proof): saturated MQAR T=512/K=128, WM-off 0.500 → WM-on 0.855 (+35.5 pp mean, 5-seed). WM is load-bearing **only** when the recurrent state is saturated. See [[working-memory]].
2. **"recall=0" was a probe artifact** ([[broken-probe-lessons]]): the in-run probe used the SFT `# {flatten}` format, OOD for a pretrain ckpt → 0.000; training-matched format → 0.438. Recall actually works (~44 % at 1 B).
3. **Single-binding recall has zero headroom**: the 287 M base recalls 100 % no-think at every distance — a delta-rule state trivially holds ONE binding. Can't show a WM lift where there's no headroom.
4. **Gradient-disconnect (the mechanistic root cause)**: WM injects only at think positions; think targets are −100; state-readonly zeroes the write; the injection lands on the final hidden (lm_head only), never back into the trunk. The recall target is an *emit* position → its gradient flows through the **recurrence**, never through WM. So the recurrence learned recall; WM got no recall gradient and `read_alpha` decayed 1.0 → 0.081. The read became **recency**, not content-addressable.
5. **Pretrain stream had no answer supervision**: the recall stream fed `problem_prompt` only → the bound value was never a recall target → no content-recall gradient anywhere (v10 AND v11). Fix = `text_field: [problem_prompt, qwen_completion]`.
6. **Addressing is the wall, and the trunk must be addressable**: freeze-trunk addressing FT → mass-on-binding stays ~0.01; joint-trunk co-train → ALSO fails because the keys are non-separable (see [[key-separability]]). The decisive negative: even explicit attention-placement supervision won't converge.
7. **THE FIX (validated 100 % synthetic)**: address on **token-identity** — key the read on the variable-name INPUT-EMBEDDING window (`wm_namekey_probe.py`: top1=1.00 vs chance) + a **copy/pointer readout** over the addressed span (`wm_multitok_readout.py`: EXACT 1.00 / 1.00 / 1.00 at N=48/64/96 vs base ≈0). Larger-state control: 2×/4× d_head and DeltaProduct K=2/3 all score ~0 at N≥48 → **WM fills headroom the trunk provably can't reach, more cheaply**.
8. **Reality check — real code has little WM headroom**: on realistic `code_recall` data the **recurrence already solves value-recall ~89 %** (separable identifiers like `CACHE_SIZE` are naturally cosine-separable, unlike synthetic `vN=MMMM`). v14's first live probe: `recall_off=1.000` → zero WM headroom on the code probe. **WM's only remaining opportunity = AGENTIC recall** (baseline ~0.42, real capacity wall) + synthetic saturating multibind.

## PRODUCTIONIZED (2026-06-16) — WM recall works in FREE GENERATION, leak-free, no-harm, cross-family
The discrete-key WM is now deployable, not just teacher-forced. A 400-step **frozen-trunk output-policy SFT** (`experiments/wm_output_sft.py`; trains only `lm_head`+`copy_head`, 84.7% params frozen → WM-OFF trunk byte-identical to base) teaches the model to EMIT the recalled value. Measured myself, leak-free (score the FIRST occurrence, distances 256–1536), STRICT, free greedy generation, NO forced prefix:
- **const (`NAME = value`): WM-ON 100.0% strict vs WM-OFF 0%.**
- **setvar (agentic name-reuse): WM-ON 92.8% strict vs WM-OFF 0%** — cross-family, the agentic target.
- **No-harm:** post-SFT general-code CE did not rise (Δ≈−0.18 on a held-out slice, partly in-distribution; the Pareto invariant holds).
- Adversarial review (agent a66689) confirmed the mechanism is GENUINE (value sourced from a frozen pointer into the prompt; untied `lm_head` + WM-OFF=0 rules out memorization; train/heldout disjoint; copy offset train==eval). Two eval/robustness bugs fixed: extractor now strips wrapping quotes (setvar strict 0%→92.8%, was a pure metric artifact — the copy faithfully copies the source's trailing `"`); tied-embeddings guard; `mem_copy_require_match` None→default-True.
- decode path: validated via `eval_code_recall.generate_copy_full` (full-forward greedy that fires the copy head via a value-span `mem_read_mask`). Open follow-up: wire the copy head into the SHARED `generate`/`forward_step` (currently they pass no read mask → copy is a no-op there) for deploy-speed incremental decode. Commit `9b198e3`. Ckpt `checkpoints/_smoke_wm_sft.pt`. See [[project_recall_discrete_key_direction]].

## KEY-DESIGN INVERSION (2026-06-16) — soft continuous name-span key > discrete (hash/VQ)
Prompted by the user's question ("is the hash oddly spelling-specific? draw inspiration from attention?"), a clean 3-arm head-to-head (`wm_vqkey_probe.py`, commit 72ec39a, leak-free held-out words, identical copy readout + frozen trunk) tested HASH (discrete exact) vs VQ (discrete learned codebook, argmax) vs SOFT (continuous learned cosine), on saturation (exact spelling N=32/64/128) and robustness (case/camel surface variants):
- **SOFT wins both axes**: separability 1.00 @N=128 (matches hash) AND variant robustness (case 0.65–0.81, camel 1.00).
- HASH: separability ~1.0 but variant recall ~0 (spelling-locked). VQ: LOSES both (41% distinct-entity collision; variant ~0.01–0.37).
- **⟹ Discreteness was the mistake, not softmax.** The "soft attention diffuses to recency" WM failure was the non-separable KEY (deep hidden / contaminated fixed window), never the softmax. Given a separable **name-SPAN** key, continuous soft attention is the best addressing — adds surface-variant tolerance the discrete codes can't. See [[key-separability]] refinement.
- HONEST: probe-scale, encoder trained directly on recall; **real-model transfer is the open test** before replacing the deployed hash (the historical `mem_key_window` soft key failed because it pooled name+arrow+value, not name-span-only). DECISION (user, 2026-06-16): integrate the name-span soft key into the model + validate on the real 287M (frozen-trunk first), then pretrain.

## UPDATED VERDICT (2026-06-16) — WM REVIVED: discrete-key addressing works, integrated, Pareto-safe
The 2026-06-15 "drop WM" verdict below was **substantially a leak artifact + the wrong addressing primitive.** Two corrections (both review-confirmed, agents a05689/a490c455/aaf6efcbd):
- **The "off=1.000 → recurrence absorbs recall" was a LEAK.** The kill-gate scored the *restated* `Answer: X` (value reappeared ~47 chars earlier → trivial recency copy). Leak-free (score the FIRST occurrence, 300+ tokens after the binding): the 287M recurrence scores **0%** on real `NAME=value` recall and **44/6/0%** on synthetic multibind N=32/64/128. Real headroom exists.
- **Cosine-over-pooled-embedding addressing was the wall, not WM itself.** PROVEN: even direct attention supervision (`addr_aux=1.0`) can't move `mass_bind` off 0.014 — the pooled window dilutes the discriminating digits (`v`+digit / identifier collide). 
- **THE FIX THAT WORKS: discrete-key addressing.** Code = hash of the identifier token-SPAN (general **lexical** version: hash the alphanumeric identifier run → exact one-hot match = delta-rule with orthogonal keys, **zero cross-talk**) + the validated **copy readout**. Frozen-trunk, near-zero-shot: **mass_bind=1.0, WM-ON 100% vs recurrence 0%** on real const + synthetic N=32/64/128.
- **INTEGRATED END-TO-END (deployable, Pareto-safe) — `checkpoints/wm_discrete_v12.pt`:** vectorized extractor (5.6×), always-on read, **learned** cold-start gate (bias −6) co-trained on v12 (frozen trunk+lm_head, `read_alpha=0` frozen → copy-head-only). Result: WM-ON recall **100%** (synN64/128 + real const) vs recurrence 0%; gate **selective** (copy_g 0.9999 on answers / 0.0011 on general code); general-code CE **+0.0009 (no-harm)** = base preserved when the gate is closed. The model *learns* to use the memory autonomously and selectively.
- **HONEST SCOPE:** this is exact **name-addressing** (the query must textually reference the bound identifier — true for code/agentic name-reuse), NOT semantic recall. Covers `=`-bindings (const ✓). Gaps needing broader binding-shape detection or VQ/semantic codes: import-`as`, signature-ordinal, content-addressed `fname`, camelCase (tokenizes differently line-start vs mid-line). Agentic-family coverage being measured now.
- **NEW VERDICT: KEEP WM** — it's a real, validated, no-harm long-range recall mechanism for the name-reuse / agentic target. Next: broaden binding shapes + VQ for content-addressed; measure on agentic + a genuine N≥48 long-horizon eval. See [[project_recall_discrete_key_direction]].

## Final verdict (2026-06-15 — SUPERSEDED by the 2026-06-16 update above; kept for history)
- **WM is dead weight for the realistic workload *as we represent it*.** v14 continued v12 with the *fixed* WM stack (embedding-key addressing + copy head + answer-span `mem_read_mask`) on a ~16 %-recall mix for ~250 M tokens. WM trains itself **OFF**: `copy_g → 0.003`, `read_alpha 0.043 → 0.028` (decaying), feature-probe `wm(Δce = +0.0000)` flat the whole run, and the live code-recall probe reads **off = 1.000** (measured) with `recall_on=0.965` (WM if anything slightly *hurts*).
- **The honest scope caveat (audited 2026-06-15, fair-baselines pass):** our realistic recall data tops out at **N=24 bindings (code) / N=15 (agentic)** — see `data/code_recall_heldout.jsonl` (hist {8,12,16,20,24}) and `data/agentic_recall_heldout.jsonl` (hist {6,10,15}). WM's validated niche is **N≥48**. So `off=1.00` is **expected, not surprising** — the data never enters WM's regime. Two compounding reasons, both true at N≤24: (a) the recurrence has capacity for ≤24 separable bindings, and (b) anything you TRAIN it to recall in that range, it learns ([[route-around-principle]] at the data level). The earlier "agentic off 42 %→100 %" framing was **partly inferred** — only the *code* off=1.000 was directly measured; the agentic teacher-forced off-arm was scripted (`/tmp/agentic_wm_eval.py`) but **not yet run** (would OOM the v12 keeper run; deferred to a free GPU). Don't quote the agentic number as measured.
- **What is NOT established:** whether a *genuine* long-horizon agentic workload (a long multi-file session tracking 50+ live entities, i.e. N≥48 beyond what fits the state) would let WM help. Our generators cap at N≤24, so this is **untested**, not refuted. That is WM's only remaining plausible niche.
- **The deep "why" (holds within the tested range):** the recurrence IS a content-addressable associative memory (read = S·q) and sits directly in the gradient, so it is the low-loss path for any recall it has capacity for. WM is only the cheaper path on inputs the recurrence provably *cannot* hold (N≥48 non-memorizable keys), validated 100 % (embedding-key + copy) vs larger-state controls ≈0.
- **Recommendation: drop WM from the code model** for the current target — recurrence + [[product-key-memory]] (facts) + [[latent-thinking-verdict]] (depth) carry the headline, and WM adds ~nothing at N≤24. **Keep WM in the codebase as a documented, validated option**, and revisit it *only* if/when we build a genuinely N≥48 long-horizon agentic eval (the one untested regime). To fully close the loop: run the deferred agentic teacher-forced off-arm on a free GPU, and generate an N≥48 realistic-agentic probe.
- This whole saga is the cleanest demonstration of [[route-around-principle]] and [[objective-function-alignment]] — and a reminder ([[broken-probe-lessons]]) to quote only what was measured.
